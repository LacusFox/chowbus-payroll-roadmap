§2 用户痛点与机会点

第二幕 · 矛盾点燃 — 4 个软件来回切、纸质签字、微信对账、小费税不会算…痛点的尽头，就是 Chowbus 的舞台

用户痛点 · 现状有多痛
痛点 01
发工资要开4个软件
排班在一个系统，打卡在另一个，算工资靠 Excel，发工资在银行网站，中间全靠手动搬运数据。
「每次发工资我都要搞两三个小时，还经常搞错。」
痛点 02
线下流程无凭无据
工时靠口头确认或微信截图，工资单靠纸质签字。员工说没拿到？老板说已经给了？无据可查，纠纷难断。
「有员工事后说工资给少了，我没有系统记录，很麻烦。」
痛点 03
合规是盲区，出事才知道
小费抵扣最低工资（小费抵扣（Tip Credit））、加班双倍计算（CA）、每周必发（NY）——大多数老板不知道这些规则，违规了才被追查。
「我不知道小费算进去之后工资还要不要补差价。」
痛点 04
英文界面，华人用不惯
ADP、QuickBooks、7shifts 全是英文界面。华人老板看不懂，只能靠会计帮忙，额外花钱。竞品无一做中文本地化。
「QuickBooks 我搞不太懂，都让会计来弄，每次还要付费。」
痛点 05
税务焦虑，怕出错
联邦税、州税、季报、年报……老板和会计都怕算错被罚。现有软件给的是数据，不是指引，出了问题不知道该怎么办。
「每到报税季我就很紧张，不知道有没有漏掉什么。」
痛点 06
小费计算混乱，容易违规
Tip pooling 规则、FICA Tip Credit 退税抵免——大部分餐厅老板根本不知道有这些，更不知道怎么算。具体能省多少要看员工小费占比、工资结构和历史申报情况，需 CPA 确认。
「小费怎么分配我是按经验来的，不确定合不合法。」
机会点 · Chowbus 能做什么
🔗
数据已在 Chowbus，打通最后一公里
排班、打卡、小费数据已有，Payroll 是自然延伸。无需用户重复输入，直接打通发薪全流程。
竞品无法复制的数据优势
🇨🇳
中文原生 workflow，零竞争蓝海
主流 Payroll/POS 厂商至多有零散中文帮助文档，没有为北美华人餐厅设计的中文原生工作流。Chowbus 服务华人餐厅多年，语言 + 信任双重优势，先发占领这条赛道。
市场空白，先发优势
🚨
AI 合规护航，而非数据堆砌
规则引擎硬性检测 Tip Credit 缺失、加班可疑、最低工资差额，AI 把每条规则的来源和操作建议翻译成中文——把专业知识变成傻瓜式引导。最终判断仍由 CPA 拍板，但 80% 的合规盲区在系统层就被点亮。
差异化体验卖点
💰
小费税抵免（FICA Tip Credit），帮老板省真金白银
餐厅老板为员工小费缴的「社保 + 医保税」共 7.65%，可向美国国税局申请退抵。一家年小费 20 万美金的餐厅，每年潜在退税 1 万美金起。系统自动识别有资格的员工 + 一键生成给会计师用的「资料表」（IRS 小费税抵免申请表 Form 8846），具体退多少由会计师审核签字。
可量化的真金白银 · 直接成为销售卖点

§6 风险一览

第六幕 · 戏外彩排 — 开演前把意外都摸清楚 · 6 道关卡 + 应对剧本，哪些坑要避、哪些雷不踩

⚠️ 关键风险点
⚠️
高风险
责任边界模糊 · 用户依赖系统结果
MVP 一旦展示工资单 + 扣税估算 + 员工签收，老板和员工天然会把它当成 Chowbus 的"最终结论"。即使加 disclaimer，少发 / 算错 / 扣税错也很难完全切割责任。
应对策略
产品命名为 薪资准备（不替代正式薪资系统） / Pay Run Assistant 而非 Payroll System；工资单永远标"草案 · DRAFT"+ 会计师复核字样；用户协议明确"Chowbus 不承担 tax filing 责任"；员工签字定位为"争议记录"不是"工资发放凭据"；P2 接 嵌入式薪资合作方（Gusto Embedded / Check 这类专业薪资公司）后由伙伴承担税务责任。
🤝
高风险
嵌入式薪资合作方（Gusto Embedded / Check 这类专业薪资公司）谈判失败
Gusto Embedded / Check 等 BaaS 不一定愿意接餐饮高复杂场景（小费池、cash tips、tip credit、多门店）。商务谈判、KYB/KYC、责任划分、co-branding 各环节都可能卡住，导致 P2 没有可靠的税务能力底座。
应对策略
MVP 阶段同步启动 商务合作评估（BD），候选至少 3 家（Gusto Embedded / Check / 第三家）并做 POC；明确合作条款（payroll of record 归属、tip 场景支持、revenue share、support 分工）；如所有候选均不可行，则 P2 仅做 CPA packet 加强 + 接 Symmetry/Vertex 等合规数据，把 ACH/税务交回 CPA。
🔗
高风险
跨模块依赖 & 数据质量
Payroll 强依赖打卡/小费池（按规则在员工间分配）/员工管理模块的数据质量。如果上游数据缺失或格式不标准，Payroll 计算结果就不可信。
应对策略
早期与相关 PM 对齐数据接口规范；Payroll 模块内置数据校验层，异常数据弹窗提示而非静默通过；定义清晰的模块边界和责任归属。
💰
中风险
小老板不愿付费
华人小馆子价格极度敏感。单店 5-15 人规模，月均省下的时间成本未必能说服他们为 Payroll 功能额外买单。
应对策略
MVP 阶段打包进现有 Chowbus 套餐，降低决策门槛；FICA Tip Credit 等抵免要做"线索发现 + CPA 备料"包装，强调"可能可观，需 CPA 确认"，不能承诺具体金额；连锁 / 集团客户（付费决策者是会计而非老板）作为主要商业目标。
🏃
中风险
竞品追加中文支持
Toast / 7shifts / Homebase 若意识到华人市场并推出中文界面，Chowbus 的语言入门壁垒将削弱。
应对策略
语言只是入门壁垒，真正护城河是「排班 + 打卡 + Tip + Payroll 全链路数据打通 + 亚裔餐厅垂直合规（ITIN / 现金小费高占比 / 家族雇佣）」—— 这些场景细节竞品需要时间和本地化团队才能复制，尽快夯实并建立品牌心智。
🚢
低风险
6 个月 MVP 范围蔓延
Payroll 功能天然复杂，开发过程中极易把 2.0/3.0 功能拉入 MVP，导致延期上线。
应对策略
严格锁定 MVP 边界：不做实际 ACH 打款 · 不做税务代申报 · 不承诺最终扣税准确性 · 只支持 5 个州基础规则。超出边界的需求统一进 Backlog，季度评审时再排期。MVP 验收硬指标：完成 6 步全自动跑薪 + 入职合规 + CPA packet 导出即上线。
